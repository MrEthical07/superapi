# SuperAPI v0.8.0 — Refactor & Upgrade Design

Status: **Draft for review** · Target release: **v0.8.0** (current baseline v0.7.x) · Author: paired design pass

This document is the end-to-end plan for the v0.8.0 sweep. It is meant to be reviewed before any code changes. It captures the current state, the agreed decisions, the exact per-file changes each phase requires, the goAuth v0.4.0 integration surface, and the verification loop.

Nothing in this document has been applied to the codebase yet.

---

## 1. Goals

The sweep resolves the structural issues that have accumulated in the template and lands the goAuth v0.4.0 upgrade:

1. **Kill the sqlc-vs-store duality.** Standardize the data layer on **sqlc**; collapse the bespoke store/operations abstraction to a thin transaction boundary.
2. **Make the document/NoSQL store optional and deletable**, not a mandatory noop wired into core.
3. **Decouple tenancy** behind a single config flag with clean seams, so a non-multi-tenant user can disable it with a switch (and delete it cleanly).
4. **Adopt goAuth v0.4.0 fully** — remember-me + session ceiling, hybrid-mode fixes, graceful logout, sliding-window limiting, Ed25519 key rotation, and **WebAuthn scaffolded but off by default**.
5. **Slim the DevX generators** so they emit the one sqlc-based pattern instead of the old store pattern and a duplicate provider.
6. Fix accumulated smells and ship v0.8.0 (docs, CHANGELOG, quality gates).

### Agreed decisions (locked)

| Area | Decision |
|---|---|
| Data layer | **sqlc under repositories**; store collapses to a thin `WithTx`/pool boundary |
| Document store | **Optional, self-contained, deletable** package; removed from core wiring |
| Tenancy | **Config flag + clean seams** (default proposed: off — see §8.1) |
| goAuth v0.4.0 | **Full adoption**; WebAuthn **scaffolded, disabled by default, no forced schema** |
| DevX tooling | **Slim to match the new data layer** |

## 3. Current-state findings (what we're fixing)

**Data layer duality is real.** The runtime auth path (`internal/core/auth/user_repository.go`) executes hand-written SQL strings through a custom `storage.RelationalOperation` abstraction. The generated sqlc package (`internal/core/db/sqlcgen/`) is imported in exactly one place — `cmd/authgen/templates.go`, a code generator — so **sqlc never runs in the actual application**. Two data-access paradigms coexist, which `AGENTS.md` §8 explicitly forbids.

**The store abstraction is barely load-bearing.** `storage.Relational*` is referenced in only three files (`deps.go`, `user_repository.go`, `modulekit/runtime.go`). No feature module uses it. `document_noop.go` + `DocumentStore` exist purely to reserve a contract for a NoSQL backend that does not exist.

**Tenancy is woven into core.** Tenant logic lives in the policy engine (`auth.go`, `presets.go`, `options.go`, `metadata.go`, `validator_rules.go`), the cache manager, and the rate limiter — but there is **no config flag**; `config.go` has zero tenant references. Tenancy is effectively always-compiled-in and only "activates" when a principal carries a `TenantID` (which the default JWT never does). Removing it today means editing many core files.

**DevX tooling is heavy and partly ceremonial.** `authgen` (~1.5k LOC) generates a `SQLCUserProvider` that differs from the real runtime provider — i.e. the generator itself embodies the duality. `modulegen` scaffolds modules on the old store pattern. `modulesync` copies module SQL to root `db/` for sqlc — a chain that currently feeds nothing at runtime.

**Smaller smells.** `internal/modules/system/routes.go` registers `whoami` twice via an `if limiter != nil` branch. Login/refresh handlers call the goAuth engine directly rather than through a service layer. There is **no logout route** at all.

## 4. Target architecture (after the sweep)

```
Handler → Service → Repository → sqlc.Queries → pgx (pool or tx)
                         │
                         └─ reads: pool-bound Queries
                            writes: tx-bound Queries via storage.WithTx(ctx, fn)
```

- **sqlc** is the single relational query engine. Generated `*sqlcgen.Queries` live **only inside repositories**; services and modules never see sqlc/pgx types (this preserves the useful half of the old `AGENTS.md` rule).
- **`storage`** shrinks to a transaction boundary: begin a pgx tx, stash it in context, and hand repositories a `*sqlcgen.Queries` bound to the tx-or-pool. `WithTx(ctx, fn)` semantics are preserved so services keep owning the write boundary.
- **Document store** is not in core. It ships as an optional, self-contained package that a user imports only if needed, and can delete otherwise.
- **Tenancy** is gated by `config.Tenancy.Enabled`; tenant policies/keying are isolated behind clean seams so disabling is one switch and full removal is a short, documented deletion.
- **goAuth v0.4.0** is wired through `internal/core/auth/config.go` with the new opt-in features; WebAuthn is present but disabled.

## 6. Phase 1 — Standardize the data layer on sqlc

This is the core structural change. It removes the duality and the bespoke execution abstraction.

> **Status: implemented (2026-07-14).** The `storage` package is now a single
> `Postgres` boundary (`Queries(ctx)` + `WithTx`); the auth repository runs on
> sqlc; `deps.go` exposes one `DB *storage.Postgres`; `modulekit.Runtime`
> exposes `DB()`; `cmd/perftoken` is migrated. The `UpdateAuthUserPasswordHash`
> query was changed from `:exec` to `:one … RETURNING id` (regenerated) to keep
> the not-found semantics (§16.2 #2). `AGENTS.md` §1/§3/§4/§6/§13/§14 and the
> directly-wrong accessor references in `docs/modules.md` were updated in the
> same commit. Build, full tests, and `superapi-verify` all pass. The remaining
> prose doc pages that still describe the old store pattern
> (`docs/architecture.md`, `docs/module_guide.md`, `docs/workflows.md`,
> `docs/crud-examples.md`, `docs/auth-goauth.md`, `docs/auth-bootstrap.md`,
> `README.md`, `docs/overview.md`) are migrated in the Phase 6 documentation
> batch (§11) to avoid re-churning them across intervening phases.

### 6.1 What we already have (reusable)

`internal/core/db/sqlcgen/` is already generated (sqlc v1.30.0) and includes exactly the auth queries we need:

- `GetAuthUserByLogin(ctx, email string) (User, error)`
- `GetAuthUserByID(ctx, id pgtype.UUID) (User, error)`
- `CreateAuthUser(ctx, CreateAuthUserParams) (User, error)`
- `UpdateAuthUserStatus(ctx, UpdateAuthUserStatusParams) (User, error)`
- `UpdateAuthUserPasswordHash(ctx, UpdateAuthUserPasswordHashParams) error`

`sqlcgen.New(DBTX)` accepts anything implementing `Exec/Query/QueryRow` — i.e. **both `*pgxpool.Pool` and `pgx.Tx`** — and `Queries.WithTx(tx)` rebinds to a transaction. This is the hook the thin store uses.

### 6.2 New thin `storage` package

Delete the operation/executor machinery; replace with a transaction boundary that yields sqlc queries.

**Remove:**
- `storage/contracts.go` — `RelationalStore`, `DocumentStore`, `RelationalOperation`, `DocumentOperation`, `RelationalExecutor`, `RowScanner`, etc.
- `storage/operations.go` — `RelationalExec/QueryOne/QueryMany`, `DocumentRun`.
- `storage/document_noop.go` — moves to Phase 2 (optional package), removed from core.
- The executor half of `storage/postgres_store.go`.

**Keep/rewrite** `storage/postgres_store.go` to something like:

```go
package storage

type Postgres struct { pool *pgxpool.Pool }

func NewPostgres(pool *pgxpool.Pool) (*Postgres, error) { ... }

// Queries returns sqlc queries bound to the tx in ctx, else the pool.
func (p *Postgres) Queries(ctx context.Context) *sqlcgen.Queries {
    if tx, ok := ctx.Value(txKey{}).(pgx.Tx); ok {
        return sqlcgen.New(tx)
    }
    return sqlcgen.New(p.pool)
}

// WithTx runs fn inside a pgx transaction (services own this boundary).
func (p *Postgres) WithTx(ctx context.Context, fn func(ctx context.Context) error) error {
    // begin tx, ctx = context.WithValue(ctx, txKey{}, tx), fn(ctx), commit/rollback
}
```

This keeps the existing `runnerFromContext` idea (tx-in-context) but returns `*sqlcgen.Queries` instead of the custom executor. `WithTx` keeps the same panic-safe rollback logic already in `postgres_store.go`.

> Design note: repositories obtaining `Queries(ctx)` per call keeps sqlc objects out of service/module interfaces while still letting services control the tx boundary via `WithTx`. This is idiomatic sqlc + pgx and satisfies the intent of `AGENTS.md` §3/§4.

### 6.3 Rewrite the auth repository

`internal/core/auth/user_repository.go`: drop the raw SQL constants and `storage.RelationalQueryOne`. Each method calls the corresponding sqlc query via `p.pg.Queries(ctx)` and maps `sqlcgen.User` → `StoredUser`:

- `GetByIdentifier` → `GetAuthUserByLogin`
- `GetByID` → parse the string id into `pgtype.UUID`, call `GetAuthUserByID`
- `Create` → build `CreateAuthUserParams` (note: `Role` is `pgtype.Text`, `Permissions int64`)
- `UpdateStatus` → `UpdateAuthUserStatus`
- `UpdatePasswordHash` → `UpdateAuthUserPasswordHash`

Mapping helpers convert `pgtype.UUID → string`, `pgtype.Text → string` (role). `pgx.ErrNoRows` → `ErrAuthUserNotFound` unchanged.

> Behavior parity note: the old raw `CreateAuthUser` used `NULLIF($3,'')` for role and hardcoded `permissions = 0`; the sqlc `CreateAuthUserParams` takes `Role pgtype.Text` and `Permissions int64`. The repo maps an empty role to `pgtype.Text{Valid:false}` and permissions to `0` to preserve behavior. Confirm the `users` schema (`db/schema/auth_users.sql`) matches.

### 6.4 Wiring changes

- **`internal/core/app/deps.go`:** replace `storage.NewPostgresRelationalStore(pool)` with `storage.NewPostgres(pool)`. Remove `Dependencies.RelationalStore`, `Dependencies.DocumentStore`, and the `Store storage.Store` field (or replace all three with a single `DB *storage.Postgres`). Remove the `DocumentStore: storage.NoopDocumentStore{}` initializer. `auth.NewRelationalUserRepository(deps.DB)`.
- **`internal/core/modulekit/runtime.go`:** remove `Store()`, `RelationalStore()`, `DocumentStore()`. Add `DB() *storage.Postgres` for modules that need relational access. (Document access comes from the optional package in Phase 2, not here.)
- **`internal/core/auth/user_repository.go`:** constructor takes `*storage.Postgres` instead of `storage.RelationalStore`.

### 6.5 sqlc config

`sqlc.yaml` is already correct (`db/queries`, `db/schema`, out `internal/core/db/sqlcgen`, `pgx/v5`). No change unless we add MFA/WebAuthn tables (Phase 4). Regeneration is only needed when we touch `db/queries` or `db/schema`.

### 6.6 Checkpoint
```
cd API
make sqlc-generate      # only if schema/queries changed (not required for pure rewire)
go build ./...
go test ./...
go run ./cmd/superapi-verify ./...
```
Watch for: any remaining importers of the deleted `storage` symbols (grep `RelationalStore|RelationalOperation|NoopDocumentStore` should be empty except tests we update).

## 7. Phase 2 — Optional, deletable document store

**Intent:** honor "introduce a specific document store, decoupled, so a user can optionally delete it or it's eliminated by dead-code removal."

> **Status: implemented (2026-07-14).** Added `internal/storage/document` — a
> self-contained package outside `internal/core`. It defines its own `Store`
> interface (`Collection(ctx, name)` + `WithTx(ctx, fn)`), an `InMemoryStore`
> reference implementation with copy-on-write transaction staging (dependency-free;
> swap in Mongo/other by implementing `Store`), and a compiling, unregistered
> example module (`.../example`) showing the handler→service→repository pattern
> with a transactional batch write and no `if sql else mongo` branching. Core
> keeps **zero** references: verified that `go list -deps ./cmd/api` does not
> include the package, so package-level dead-code elimination holds — an unused
> document store is excluded from the binary and deleting the folder is clean.
> Added `docs/document-store.md`. Package + example tests, full build, and
> `superapi-verify` all pass.

**Design:**
- New self-contained package, e.g. `internal/storage/document/` (deliberately **outside** `internal/core`, so its optionality is visible). It defines its own small interface and (optionally) a concrete Mongo/other implementation. It does **not** import core wiring, and core does **not** import it.
- Because Go only compiles packages that are imported into the build, an unused document package is naturally excluded from the binary — "dead-code elimination" at the package level. A user who never wires it pays nothing; a user who wants it gone deletes the folder.
- Ship a **documented example module** (in docs, or a commented `examples/` module) showing how to construct the document store in the module's `BindDependencies` and use it in that module's repository — with no `if sql else mongo` branching in shared code (per `AGENTS.md` §8).
- Core keeps zero references: no `DocumentStore` field on `Dependencies`, no `DocumentStore()` on the runtime.

**Checkpoint:** `go build ./...` (the package compiles standalone; nothing in core references it).

## 8. Phase 3 — Decouple tenancy behind a config flag

**Intent:** one switch disables tenancy cleanly; full removal is a short documented deletion. No behavior surprises for people who do want tenancy.

> **Status: implemented (2026-07-14).** Added `config.TenancyConfig`
> (`TENANCY_ENABLED` default false, `TENANCY_ENFORCE_ISOLATION`) with a lint
> rule. The policy package gained a package-level tenancy flag
> (`SetTenancyEnabled`/`TenancyEnabled`) that **defaults to enabled** in its zero
> state — so unit tests and any consumer that never configures it keep strict
> behavior — and is set to `cfg.Tenancy.Enabled` at startup in
> `initDependencies`, before any route registers. Tenant policies and presets
> (`TenantRequired`, `TenantMatchFromPath`, `TenantRead`, `TenantWrite`) moved
> into `policy/tenant.go`. `defaultPresetConfig` now varies by user id (not
> tenant) and clears the tenant path param when tenancy is off;
> `validateTenantRules` only forces tenant policies on `{tenant_id}` routes when
> tenancy is on (the `TenantMatchFromPath requires TenantRequired` dependency
> rule always holds). goAuth's `MultiTenant.Enabled/EnforceIsolation` is wired
> from the flag through `auth.TenancySettings` →
> `ProjectGoAuthConfig`/`NewGoAuthEngine`. Cache and rate-limit tenant surfaces
> were left intact (they already degrade gracefully). Added `policy/tenant_test.go`
> covering both flag states, `docs/removing-tenancy.md`, the tenancy note in
> `docs/policies.md`, and the env-var section. Build, full tests (flag default on
> in unit tests, off at app startup), and `superapi-verify` all pass.

### 8.1 The flag

Add to `config.go` a `Tenancy` section:
```
TENANCY_ENABLED (bool)   // proposed default: false
```
**Open decision (see §13):** default `false` gives the cleanest out-of-the-box template and matches today's reality (default JWTs carry no tenant, so tenancy is inert anyway). Default `true` preserves the *capability* being present for existing consumers. Recommendation: **false**, documented as opt-in.

The flag flows into `Dependencies` and the goAuth config (`cfg.MultiTenant.Enabled = tenancyEnabled` — goAuth exposes `MultiTenantConfig{Enabled, EnforceIsolation}`; note goAuth tenant scoping comes from `WithTenantID` on the request context, and `TenantHeader` is a documented no-op).

### 8.2 Seams to refactor

Isolate all tenant logic so it is registered/keyed only when enabled:

- **`internal/core/policy/`**: move `TenantRequired`, `TenantMatchFromPath` (from `auth.go`) and the `TenantRead`/`TenantWrite` presets (`presets.go`) into a dedicated `policy/tenant.go`. Keep the policy *types* (`PolicyTypeTenantRequired`, etc.) but ensure they are only attached when tenancy is on.
- **`internal/core/policy/validator_rules.go`**: `patternContainsTenantID` currently *forces* tenant policies on any `{tenant_id}` route. Gate this: when tenancy is disabled, a `{tenant_id}` route is a normal param (or the validator warns). When enabled, keep the strict rule.
- **`internal/core/policy/options.go`**: `defaultPresetConfig` sets `cacheVaryBy.TenantID = true` and `tenantMatchParam = "tenant_id"` unconditionally. Make the tenant vary/tag default follow the flag (off when tenancy off) so authenticated cache routes vary by user only.
- **`internal/core/cache/manager.go`**: `VaryBy.TenantID` and tag `TenantID` remain supported, but when tenancy is off they must not be *required*; the validator rule "authenticated cache routes require VaryBy.UserID **or** TenantID" is already satisfied by UserID, so this mostly needs the option default change above. When a tenant tag is requested but no tenant is present, today it errors ("missing tenant id for cache tag") — with tenancy off that path should be unreachable by construction.
- **`internal/core/ratelimit/ratelimit.go`**: `ScopeTenant`, `KeyByTenant`, and the composite `KeyByUserOrTenantOrTokenHash` already degrade gracefully (tenant branch yields `ScopeAnon` when absent, falling through to user/token). Keep them; just ensure presets don't default to `ScopeTenant` when tenancy is off.
- **`internal/core/tenant/`**: the helper package stays; it's the natural home for tenant primitives and a clean deletion target.

### 8.3 Clean-removal story (documented)

With the above, disabling is `TENANCY_ENABLED=false`. **Deleting** tenancy entirely becomes: remove `policy/tenant.go`, the `internal/core/tenant/` package, the `Tenancy` config block, and the (now-unused) tenant fields in cache/ratelimit option structs — a bounded, greppable list we document in `docs/`.

**Checkpoint:** `go build ./...`, `go test ./...`, `go run ./cmd/superapi-verify ./...` with the flag both on and off.

## 9. Phase 4 — Full goAuth v0.4.0 adoption

All new goAuth surface is opt-in; we wire it through `internal/core/auth/config.go` (the documented customization point) plus new endpoints in the system module. **Signatures below were read from the local `Auth Guard` v0.4.0 source; re-confirm against `docs/api-reference.md` before coding.**

> **Status: implemented (2026-07-14).** Signatures were re-verified against the
> **published** v0.4.0 in the module cache. Landed:
> - **Config** (`auth/config.go` + new `auth/config_env.go`): `AUTH_MAX_SESSION_DURATION`
>   → `Session.MaxSessionDuration`; `AUTH_LIMITER_WINDOW_MODE` →
>   `Security.LimiterWindowMode`; `AUTH_KEY_ID` + `AUTH_VERIFY_KEYS` →
>   `JWT.KeyID`/`JWT.VerifyKeys` with the set-both-or-neither invariant enforced
>   early; `WEBAUTHN_*` → `WebAuthn` config (off by default); `MultiTenant` from
>   Phase 3.
> - **Login/MFA/logout** route through a new thin `system.authService`. Login uses
>   `LoginWithOptions` with `RememberMe` and returns an MFA-challenge shape when
>   required; `mfa/confirm` calls `ConfirmLoginMFAWithType`; logout calls
>   `LogoutByAccessToken` (§16.2 #3). `whoami` is now registered once.
> - **WebAuthn** scaffolded off: optional migration `000004_webauthn_credentials`
>   + schema mirror + sqlc queries; `WebAuthnCredentialRepository`; the four
>   `WebAuthnCredentialProvider` methods on `StoreUserProvider`; auth-protected
>   ceremony endpoints that return "webauthn disabled" until enabled. `Build()`
>   only requires the capability when `WebAuthn.Enabled` (verified in v0.4.0
>   `builder.go`).
> - **Hybrid audit:** no code change needed — `authRequiredWithEngine` already
>   passes `ModeInherit` for hybrid and an explicit route mode otherwise, matching
>   v0.4.0 `Guard`/`Validate`. The `ModeJWTOnly` downgrade caveat is documented in
>   `docs/policies.md`.
> - **Static verify:** route registration must pass policies directly (no variadic
>   spread); whoami and WebAuthn routes were written accordingly.
> - **Tests:** `bearerToken`, `buildLoginResponse` MFA branch,
>   `webAuthnCredentialToView` round-trip, and `config_env` (envDuration + key
>   rotation invariant). Build, full tests, and `superapi-verify` all pass.
> - **Docs:** `docs/enabling-webauthn.md` (new), env-vars sections, `policies.md`
>   hybrid guarantee + `ModeJWTOnly` caveat, and a v0.4.0 features section in
>   `docs/auth-goauth.md`. The deeper `auth-goauth.md`/`authDocs/` prose rewrite
>   remains in the Phase 6 batch.

### 9.1 Config wiring (`internal/core/auth/config.go`)

- **Session ceiling / remember-me:** set `cfg.Session.MaxSessionDuration` from a new env (e.g. `AUTH_MAX_SESSION_DURATION`, unset → goAuth per-mode default). Field confirmed: `SessionConfig.MaxSessionDuration time.Duration` (validation rejects negative and `(0,1m)`).
- **Sliding-window limiting:** set `cfg.LimiterWindowMode = "sliding"` when a new env opts in (e.g. `AUTH_LIMITER_WINDOW_MODE=sliding`; goAuth validates `"fixed"|"sliding"`, default `"fixed"`). This governs goAuth's internal auth-abuse limiter. (SuperAPI's *own* `internal/core/ratelimit` limiter is separate; mirroring a sliding window there is a larger, optional follow-up — noted, not in scope.)
- **Ed25519 key rotation:** populate `cfg.JWT.VerifyKeys map[string][]byte` (kid → public key) from env/secret when rotating, and `cfg.JWT.KeyID` for the active signing key. Document the overlap-window procedure and the `goauth-keygen` CLI in `docs/`.
- **Multi-tenant:** `cfg.MultiTenant.Enabled` follows the Phase 3 flag.
- **WebAuthn:** set `cfg.WebAuthn` from env, **`Enabled` default false** (see 9.4).

### 9.2 System module endpoints (`internal/modules/system/`)

- **Login → remember-me + MFA-aware.** Switch `m.login` from `engine.Login(...)` (returns `access, refresh, err`) to `engine.LoginWithOptions(ctx, id, pw, goauth.LoginOptions{RememberMe: req.RememberMe})` which returns `*goauth.LoginResult`. Add `RememberMe bool` to `authLoginRequest`. Read tokens + MFA fields from `LoginResult` (confirm field names: `AccessToken`, `RefreshToken`, `MFARequired`, `MFAType`, `MFATypes`). When `MFARequired`, return an MFA-challenge response instead of tokens.
- **MFA confirm endpoint (scaffolded).** New route calling `engine.ConfirmLoginMFAWithType(ctx, ..., "totp"|"webauthn")`. Wire it but keep TOTP/WebAuthn disabled by default so it's an example.
- **Logout endpoint (new).** There is currently **no logout route**. Add `POST /api/v1/system/auth/logout` that passes the access token to goAuth's logout — v0.4.0 accepts an **expired-but-valid** access token gracefully, so no special-casing. This also fixes a real gap in the template.
- **`whoami` smell fix** folded in here or Phase 6: collapse the `if limiter != nil` double-registration into a single registration that conditionally includes the rate-limit policy.

### 9.3 Hybrid mode

SuperAPI enforces auth via `policy.AuthRequired(engine, mode)`, not goAuth's `middleware.*`. v0.4.0 adds `middleware.RequireHybrid(engine)` and fixes `Guard(engine, ModeHybrid)`. Action: audit `policy/auth.go` to ensure our hybrid path matches the now-documented guarantee ("Hybrid validates JWT-only unless a route opts into `ModeStrict`; explicit per-route mode wins"). Respect the **security caveat**: a per-route `ModeJWTOnly` downgrade bypasses revocation/version/device/account-status checks — only downgrade routes safe to serve from the JWT alone. Document this in `docs/policies.md`.

### 9.4 WebAuthn — scaffolded, off by default, no forced schema

Per the decision: provider + config + endpoints present, disabled by config, **without forcing the schema** on users who don't enable it.

- **Provider capability:** implement `goauth.WebAuthnCredentialProvider` on `StoreUserProvider` (methods: `GetWebAuthnCredentials`, `AddWebAuthnCredential`, `UpdateWebAuthnCredentialSignCount`, `RemoveWebAuthnCredential`). Backed by a `webauthn_credentials` table + sqlc queries.
- **No forced schema:** ship the `webauthn_credentials` migration as a **separate, clearly-labeled optional migration** that operators apply only when enabling WebAuthn. Until then `cfg.WebAuthn.Enabled = false`, `Build()` does not require the capability, and the provider methods are inert. (`Build()` only fails closed when WebAuthn is enabled *and* the capability is missing — so shipping the methods is safe and enabling is a config+migration step.)
- **Config:** `WebAuthnConfig{Enabled, RPID, RPDisplayName, RPOrigins, AttestationPreference, UserVerification, CeremonyTTL, RequireForLogin, RejectClonedAuthenticators}` from env, all gated on `Enabled=false` by default.
- **Ceremony endpoints (registered, gated):** thin handlers relaying JSON to/from the browser WebAuthn API:
  - `BeginWebAuthnRegistration(ctx, userID) (*WebAuthnRegistrationChallenge, error)` / `FinishWebAuthnRegistration(ctx, userID, ceremonyID, responseJSON) (*WebAuthnCredential, error)`
  - `ListWebAuthnCredentials(ctx, userID)` / `RemoveWebAuthnCredential(ctx, userID, credentialID)`
  - `BeginWebAuthnLogin(ctx, challengeID) ([]byte, error)`, confirmed via `ConfirmLoginMFAWithType(..., "webauthn")`.
  - New Redis prefix `awn:` is managed by goAuth (single-use ceremonies) — no work for us.

## 13. Open decisions & risks

**Decisions — now settled (2026-07-14):**
1. **Tenancy default → `TENANCY_ENABLED=false`.** Opt-in, cleanest template; matches today's inert reality. The flip of the effective default is documented as a behavior change in CHANGELOG.
2. **`storage` surface → single `DB *storage.Postgres`.** Collapse `Store/RelationalStore/DocumentStore`; the document store is its own optional package (Phase 2).
3. **authgen scope → scaffold config/roles/routes only**, no parallel sqlc provider (removes the generator-embodied duality).
4. **Auth service layer → introduce the thin `system` service now.** Login/refresh/logout/MFA all route through it, since Phase 4 adds logout + MFA anyway; improves example fidelity to the enforced architecture.

**Execution mode:** implement **one phase per iteration, commit, then pause** for the local build/test/verify checkpoint before the next phase (no in-sandbox Go toolchain).

**Risks:**
- **No in-sandbox compilation** → all verification is on your machine; expect a few compile-fix round-trips per phase, especially Phase 1 (widest blast radius) and Phase 4 (new surface).
- **sqlc regen** requires sqlc v1.30.0 locally (`make sqlc-generate`).
- **goAuth published** → API requires `v0.4.0` from the proxy directly; no `replace`, no pre-release cleanup, CI resolves it normally.
- **Behavior changes from v0.4.0 §9** — low exposure for us (no `middleware.Guard` use; MFA/webauthn off), but the hybrid-mode and MFAType-preference notes matter once MFA is enabled.
- **WebAuthn "no forced schema"** hinges on shipping its migration as optional and keeping `Enabled=false`; verify `Build()` doesn't require the capability when disabled (it doesn't, per the v0.4.0 capability-detection contract — re-confirm).

## 15. Repository / working rules for this sweep

These are the operating rules for the whole v0.8.0 sweep. They govern branch, commits, and docs — not the design itself.

- **Single branch.** All changes live on `sweep/v0.8.0-sqlc-tenancy-goauth-v0.4.0`. No work happens on `main`.

- **Feature-grained commits.** Each phase (and each self-contained sub-change within a phase) is its own commit, so history reads as the phase plan does. Prefer several focused commits over one mega-commit.

- **Commit messages are detailed and human.** Each message explains what changed and why, in specifics. **No mention of any AI agent, no co-authors, no `Co-Authored-By` / `Generated with` trailers.** Plain authored history.
- **Do not push.** Commits stay local on the branch until explicitly cleared to push to origin.
- **Branch name** signals the sweep: data-layer (sqlc) + tenancy + goAuth v0.4.0 under a `sweep/v0.8.0-…` prefix.

The `users` schema (`db/schema/auth_users.sql`) matches the mapping in §6.3 (`role TEXT` nullable, `permissions BIGINT NOT NULL DEFAULT 0`).

### 16.2 Corrections the plan needs (found during scan)

1. **`cmd/perftoken/main.go` is a Phase-1 blast-radius file the §12 inventory omits.** It calls `storage.NewPostgresRelationalStore(pgPool)` directly. When the relational store is replaced by `storage.NewPostgres`, perftoken must be migrated too, or the build breaks. **Action:** add `cmd/perftoken/main.go` to the Phase 1 row of §12.

2. **`UpdatePasswordHash` parity bug in the sqlc swap (real behavior regression).** The current raw query is `... RETURNING id::text` and the repo maps `pgx.ErrNoRows → ErrAuthUserNotFound`, so updating a **nonexistent** user surfaces "not found". The generated `UpdateAuthUserPasswordHash` is **`:exec`** — it returns `nil` on zero rows affected. Swapping to it silently changes behavior: a password reset for a missing/deleted user would report **success**. This flows up through `StoreUserProvider.UpdatePasswordHash` → goAuth's password-reset/change path. **Action (pick one), in Phase 1:**
   - change the query to `-- name: UpdateAuthUserPasswordHash :one … RETURNING id` and treat `pgx.ErrNoRows` as `ErrAuthUserNotFound` (preserves exact behavior, needs `make sqlc-generate`), **or**
   - keep `:exec` but have the repo inspect `CommandTag.RowsAffected()==0 → ErrAuthUserNotFound`. Note `:exec` discards the tag, so this actually requires an `:execrows` variant — which again means a query change + regen.
   The `:one … RETURNING id` route is the cleaner parity-preserving fix. Either way this is **not** a pure rewire; Phase 1 *does* touch `db/queries` and requires `make sqlc-generate` (contradicting §6.5's "no change" note for the pure-rewire case).

3. **goAuth logout method name is wrong in §9.2.** There is no `engine.Logout(accessToken)` that takes an access token. Verified against local v0.4.0 source (`engine.go`):
   - `Logout(ctx, sessionID string)` — takes a **session ID**, not a token.
   - `LogoutByAccessToken(ctx, tokenStr string)` — **this** is the access-token path, and it's the one with the "expired-but-authentic token accepted, returns nil" behavior the doc describes. Invalid signature/claims → `ErrTokenInvalid`.
   **Action:** the logout handler must call `engine.LogoutByAccessToken(ctx, token)`. Update §9.2 accordingly.

4. **Config field paths in §9.1 are partly wrong.** Verified against v0.4.0 `config.go` (top-level `Config` nests `JWT`, `Session`, `Security`, `MultiTenant`, `WebAuthn`, `Result`, `Account`):
   - `LimiterWindowMode` is **`cfg.Security.LimiterWindowMode`**, not `cfg.LimiterWindowMode`. Validated to `"fixed"|"sliding"` at Build.
   - `MaxSessionDuration` is `cfg.Session.MaxSessionDuration` ✓ (as written); validation: `>= 0`, and if `> 0` then `>= 1m`.
   - `VerifyKeys` is `cfg.JWT.VerifyKeys map[string][]byte` ✓, but note the **Build-time coupling**: setting `VerifyKeys` *requires* `cfg.JWT.KeyID` to be non-empty **and** present as a key in the map, else Build fails. So key-rotation env wiring must set both or neither — document that invariant.
   - `MultiTenant` is `cfg.MultiTenant{Enabled, EnforceIsolation}` ✓.
   **Action:** fix the field path for `LimiterWindowMode` and add the `VerifyKeys`+`KeyID` invariant to §9.1.

5. **go.mod baseline is `v0.3.2`, not `v0.3.0`.** §5/§12 imply bumping from a v0.3.x line; the actual `require` is `github.com/MrEthical07/goAuth v0.3.2`. Minor, but the Phase 0 bump line and CHANGELOG should say `v0.3.2 → v0.4.0`.

6. **Resolved — no `replace`, use published v0.4.0.** The earlier concern about pinning a local `../Auth Guard` checkout (which also carries later `v0.9.x-security-freeze` tags) is moot: v0.4.0 is published on the proxy and the API requires it directly. This removes the space-in-path risk and the "remove replace before release" chore entirely.

7. **WebAuthn signatures in §9.4 check out** against `engine_webauthn.go` and `types.go`: `WebAuthnCredentialProvider` has exactly the four methods listed; `BeginWebAuthnRegistration/FinishWebAuthnRegistration/ListWebAuthnCredentials/RemoveWebAuthnCredential/BeginWebAuthnLogin` exist with the stated shapes; `Build()` fails only when `WebAuthn.Enabled && provider missing`. The "scaffold + off by default + optional migration" plan is sound as written. `StoreUserProvider` already implements the TOTP/backup-code capability surface as stubs, so adding the WebAuthn methods is additive.

### 16.3 Additional smells found (not in §3, worth folding into Phase 6)
- **`StoreUserProvider` read methods use a detached 3s `context.Background()` timeout** (`lookupContext()` in `provider_store.go`) rather than the caller's request context. Under goAuth's `UserProvider` interface the read methods are context-less, so this is partly forced — but it means auth lookups ignore request cancellation/deadlines and always get a fixed 3s budget. Note it; don't necessarily change it (interface-constrained).
- **Auth handlers bypass a service layer** (login/refresh call the engine directly in `routes.go`). Already noted in §11 as optional; the logout + MFA additions in Phase 4 are a good moment to introduce a thin `system` service so all four auth flows are symmetric. Recommend doing it (open decision §13.4).
- **`config.go` validation has no home for the new envs yet** — `Lint()` is exhaustive and per-env; the Phase 3/4 envs (`TENANCY_ENABLED`, `AUTH_MAX_SESSION_DURATION`, `AUTH_LIMITER_WINDOW_MODE`, `WEBAUTHN_*`) each need a matching `lint*Env` guard to stay consistent with the file's established pattern, or they'll be the only unvalidated envs in the config.

**Cross-cutting impacts:**

- **Behavior changes that ship (must be in CHANGELOG as such):** (a) tenancy default flipping to off changes cache key variation and validator strictness for `{tenant_id}` routes; (b) the `UpdatePasswordHash`-not-found path (must be *preserved*, per §16.2 #2, else it's an unintended regression); (c) new logout endpoint; (d) login response gains remember-me/MFA shape.

Logout via `LogoutByAccessToken` accepting expired tokens is intended and safe. WebAuthn-off must be verified so `Build()` doesn't demand the capability.
